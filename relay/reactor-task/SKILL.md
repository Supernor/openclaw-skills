# Skill: reactor-task
Description: Handle Telegram-driven task planning and execution via the Reactor (Claude Code). Detects work intent, generates a plan via claude --print, presents it with action buttons, and manages the full execute/stop/undo lifecycle. All communication stays on Telegram.

## Triggers
- "build ...", "fix ...", "create ...", "implement ...", "set up ...", "refactor ..."
- "/reactor", "/build", "/plan"
- Any work request that needs host-level code execution (file edits, script creation, system changes)
- NOT triggered by: conversation, questions, status checks, skill invocations, chart commands

## Flow

### Phase 1: Detect & Plan
When Robert sends a work request:

1. Acknowledge immediately:
   > "Planning that for you..."

2. Create an ops.db task for the host executor:
   ```
   Tool: tasks_create
   agent: relay
   task: "reactor-plan: {Robert's request}"
   meta: {
     "host_op": "reactor-plan",
     "subject": "{short title for the task}",
     "description": "{Robert's full request text}",
     "telegram_chat_id": "{Robert's chat ID}"
   }
   urgency: routine
   ```

3. The host-ops-executor will:
   - Run `claude --print` with the request
   - Format the plan output
   - Send the plan to Telegram with [Go] [Tweak] [Stop] buttons

### Phase 2: Callback Handling
When Robert taps a button or replies:

**callback_data: `reactor_go_{plan_id}`**
1. Acknowledge: "Submitted! You'll get updates as it progresses."
2. Create ops.db task:
   ```
   meta: {
     "host_op": "reactor-execute",
     "plan_id": "{plan_id}",
     "telegram_chat_id": "{chat_id}"
   }
   ```

**callback_data: `reactor_tweak_{plan_id}`**
1. Ask: "What would you change?"
2. Wait for Robert's reply
3. Create a new reactor-plan task with the modified description
4. The old plan_id is discarded; a new plan is generated

**callback_data: `reactor_skip_{plan_id}`**
1. Acknowledge: "Plan cancelled."
2. No further action

### Phase 3: Control Commands (during execution)
Robert can send these messages at any time while the reactor is running:

| Message | Action |
|---------|--------|
| "stop" or "cancel reactor" | Create task: `host_op="reactor-stop"` |
| "undo" or "revert" | Create task: `host_op="reactor-undo"` |
| "status" or "reactor status" | Create task: `host_op="reactor-status"` |

For each control command:
1. Acknowledge immediately
2. Create the ops.db task with `telegram_chat_id` in meta
3. The host executor handles execution and Telegram notification

### Progress Updates
The relay-handoff-watcher automatically sends Telegram notifications for:
- **pickup**: "Working on: {subject}"
- **progress**: "Progress: {N} steps completed..." (debounced, max 1/min)
- **done**: Full completion summary
- **fail**: Error details

You do NOT need to poll or check progress — it's push-based via reactor.jsonl events.

## Important Notes
- The host executor polls every 30s, so there's a brief delay between task creation and plan generation
- Plans expire after 24 hours — don't execute stale plans
- Only one reactor task can run at a time (bridge-reactor.sh is single-threaded)
- The plan is generated by Claude Code Opus via `claude --print` — it produces better plans than Codex
- All Telegram messages use buttons via the `message` tool with `buttons` parameter
